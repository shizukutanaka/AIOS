# AI特化Linux OS 仕様書

- 文書名: AI特化Linux OS 仕様書
- 版数: v1.0-draft
- 作成日: 2026-04-09
- 文書種別: 概念仕様書 / 製品要求仕様書
- 対象読者: プロダクト企画、OS基盤開発、MLOps、インフラ運用、セキュリティ担当

## 1. 目的

本仕様書は、AI推論およびAI開発に特化したLinuxベースOSの機能要件、非機能要件、アーキテクチャ方針、実装優先順位を定義するものである。

本OSは、一般向けデスクトップOSではなく、ローカルLLM、画像生成、埋め込み生成、推論API提供、軽量なAI開発環境を安定して提供することを目的とする。

本OSの中核価値は次の2点とする。

- AI環境構築の複雑さをOSレベルで吸収すること
- GPU/NPU/NVMeを含むハードウェア性能を安全かつ再現性高く引き出すこと

## 2. 適用範囲

本仕様の適用範囲は以下とする。

- 単一ノードでのAI推論基盤
- ローカルLLMサーバー
- 画像生成ワークロード
- 埋め込み生成およびRAG補助処理
- コンテナ化されたAI開発実行環境
- ヘッドレスAIサーバー
- AIワークステーション

本仕様の適用範囲外は以下とする。

- 大規模分散学習クラスタの完全なオーケストレーション
- 一般ユーザー向けフルデスクトップ環境
- 汎用業務アプリケーションOS
- モバイル端末向けOS

## 3. 基本方針

本OSは以下の方針で設計する。

- ホストOSは最小構成とし、AIアプリケーションは原則コンテナで実行する
- ベースOSはImmutableまたはそれに準ずる更新モデルを採用する
- OS更新とAIアプリ更新を分離し、障害切り分けを容易にする
- cgroup v2、PSI、OOM制御を前提に、AIジョブを制御可能なOSとする
- GPU/NPUサポートは「全部入り」ではなく、ハードウェア別プロファイルで最適化する
- ローカル推論APIは標準提供するが、外部公開は初期状態で無効とする

## 4. 製品コンセプト

本OSは「Linux上でAIを動かす」のではなく、「Linuxを土台にしたAI実行基盤」として再定義する。

したがって、価値の中心は以下に置く。

- ドライバ導入の容易さ
- 依存関係の破綻防止
- モデル展開の再現性
- ローカル推論の起動しやすさ
- 運用時の観測可能性

## 5. 製品構成

本OSは、以下の3層で構成する。

### 5.1 Core Layer

OS共通基盤として以下を提供する。

- Linuxカーネル
- init / service管理
- コンテナ実行基盤
- 更新・ロールバック機構
- ログ・監視基盤
- 認証、権限、セキュリティ設定
- AI実行用ランチャーおよび標準CLI

### 5.2 Driver Layer

ハードウェア依存領域として以下を提供する。

- NVIDIA GPU向けドライバおよびコンテナ連携
- AMD GPU向けROCm系ドライバおよびデバイス連携
- Intel GPU / NPU向け実行環境
- AMD Ryzen AI NPU向け実行環境

### 5.3 Application Layer

アプリケーション層では以下を提供する。

- OpenAI互換APIサーバー
- ローカルLLMサーバー
- 画像生成実行環境
- 埋め込み生成エンドポイント
- Web UI
- CLI管理ツール
- 署名済みリファレンスコンテナイメージ

## 6. ハードウェアプロファイル

単一OSイメージですべてのアクセラレータを完全最適化することは、2026-04-09時点では現実的ではない。そのため、v1では共通コアとハードウェア別プロファイルを採用する。

### 6.1 Server-GPU プロファイル

対象:

- NVIDIA GPU搭載サーバー
- AMD GPU搭載サーバー
- AI推論用ワークステーション

用途:

- LLM推論
- 画像生成
- OpenAI互換API提供
- 高負荷バッチ推論

### 6.2 Edge-NPU プロファイル

対象:

- Intel Core Ultra NPU対応機
- AMD Ryzen AI対応機

用途:

- ローカルアシスタント
- 低消費電力推論
- オフライン音声/テキスト支援
- 軽量モデルの常駐利用

### 6.3 v1 のサポート方針

- v1で最優先とするのは `Server-GPU` とする
- `Edge-NPU` は限定サポートとして提供する
- Linux対応条件が厳しいNPU構成は、対応機種一覧方式で運用する

## 7. 機能要件

### 7.1 OS基盤要件

- 64bitアーキテクチャを前提とすること
- cgroup v2 を有効化すること
- PSIを利用可能とすること
- `systemd-oomd` 相当のOOM事前制御機構を持つこと
- ロールバック可能な更新機構を持つこと
- 更新単位はイメージまたはトランザクショナル更新とすること

### 7.2 ストレージ・I/O要件

- NVMeを主要ストレージとして想定すること
- 大規模モデルロードの高速化を考慮したI/O設計であること
- 非同期I/Oを前提にしたモデルロード手段を持つこと
- モデルキャッシュ領域を分離できること
- ログやキャッシュによる過剰書き込みを抑制できること

### 7.3 コンテナ基盤要件

- Podmanを標準搭載すること
- rootless実行を標準とすること
- OCIイメージ互換であること
- GPU/NPU利用時のデバイス割当手段を標準化すること
- 署名済みリファレンスイメージを配布できること

### 7.4 AIランタイム要件

- OpenAI互換APIサーバーを標準提供できること
- LLM推論サーバーをローカルで起動できること
- 埋め込み生成用エンドポイントを提供できること
- 画像生成系ワークロードをコンテナで実行できること
- モデルファイル格納場所を共通管理できること
- コンパイル済みモデルやキャッシュを永続化できること

### 7.5 開発環境要件

- Python実行環境を最小限提供すること
- `uv` 相当の高速パッケージ管理を利用可能とすること
- AIライブラリはホスト全面プリインストールではなく、コンテナ配布を基本とすること
- SSHまたはWeb UI経由で開発可能とすること
- CLI経由でコンテナテンプレートを起動できること

### 7.6 API要件

- localhost向けローカルAPIを標準提供すること
- OpenAI互換APIでの推論呼び出しを可能とすること
- APIの外部公開は初期状態で無効とすること
- 外部公開時はTLS、認証、アクセス制御を必須とすること
- 同時実行数、タイムアウト、最大トークン数を制御可能とすること

### 7.7 監視要件

- CPU、RAM、VRAM、ディスクI/O、温度、電力、利用率を監視可能とすること
- tokens/sec、推論待機時間、モデルロード時間を可視化すること
- ジョブ別、コンテナ別のリソース消費を追跡できること
- CLIとWeb UIの双方で主要メトリクスを確認できること
- ベンダー別監視情報を統合表示できること

### 7.8 ログ要件

- 監査ログ、システムログ、アプリログを区分すること
- デフォルトでは高頻度デバッグログを無効とすること
- ログ保存期間と容量上限を制御できること
- 障害解析用の詳細ログを一時的に有効化できること

## 8. 非機能要件

### 8.1 性能要件

- 起動後すぐにAIジョブを投入できる初期状態であること
- 単一モデルの初回ロード時間を短縮できる構成であること
- 同一モデル再利用時にキャッシュを活用できること
- ホストOSの常駐負荷を最小化すること

### 8.2 安定性要件

- OS更新に失敗した場合でも前版へ戻せること
- AIアプリ障害がOSコアを破壊しないこと
- ドライバ更新とアプリ更新の切り分けが可能であること
- 長時間運転時にメモリ圧迫とOOMを制御できること

### 8.3 セキュリティ要件

- 最小権限を原則とすること
- 不要ポートを開放しないこと
- rootlessコンテナを標準とすること
- 外部公開APIには認証を必須とすること
- 署名済みイメージ利用を推奨または必須とすること

### 8.4 保守性要件

- ハードウェア別プロファイルを独立更新できること
- 標準コンテナテンプレートを差し替え可能であること
- 設定をコードまたは宣言的ファイルで管理できること
- 障害切り分け時に「OS」「ドライバ」「コンテナ」「モデル」の層を明確に分離できること

## 9. 不要機能

AI実行専用OSとして、以下は標準搭載対象外とする。

- GNOME、KDE等の重量級デスクトップ環境
- 印刷サービス
- 常時有効なBluetooth機能
- 一般向けマルチメディア機能
- 32bit互換レイヤー
- オフィスソフト
- メールクライアント
- 一般用途ブラウザのフルプリインストール
- 汎用的な常駐バックグラウンドサービス
- AI用途に不要な高頻度ログ出力

必要な場合は、別コンテナまたは別端末で補う前提とする。

## 10. 推奨実装方針

### 10.1 ベースOS

以下の特性を持つOS構成を推奨する。

- Immutableまたはトランザクショナル更新に対応
- イメージベース更新に対応
- ロールバック可能
- コンテナ中心運用に適する

### 10.2 コンテナ戦略

AIライブラリ群はホストに直接展開せず、用途別コンテナとして配布する。

例:

- LLM推論用
- 画像生成用
- 埋め込み生成用
- Python開発用
- ベンチマーク用

### 10.3 API戦略

`Server-GPU` では OpenAI互換APIを基準インターフェースとする。これにより、上位アプリケーションが推論バックエンドの差異を吸収しやすくなる。

`Edge-NPU` では localhost専用APIを基本とし、UIアプリやローカルエージェントと連携させる。

### 10.4 モデル管理戦略

- モデル格納先は専用ディレクトリに統一する
- モデルのバージョン、量子化形式、用途ラベルを保持する
- ランタイム別キャッシュを永続化する
- コンパイル済みモデルやテンソルキャッシュの再利用を優先する

## 11. リソース制御方針

AIワークロードでは、単純なCPU使用率や空きメモリだけでは健全性を判断できない。そのため、以下の制御を前提とする。

- ジョブ種別ごとの cgroup テンプレートを用意する
- `memory.high` を使って急激なメモリ圧迫を抑える
- `memory.max` により暴走ジョブの上限を設定する
- PSIにより逼迫状態を検知する
- OOM前に優先度の低い推論ジョブを整理できること
- スワップは全面無効ではなく、限定利用方針を持つこと

## 12. 観測可能性要件

本OSでは、一般的なサーバー監視に加え、AI固有の指標を必須とする。

必須指標:

- モデルロード時間
- 初回推論待機時間
- tokens/sec
- バッチサイズ
- KVキャッシュ使用状況
- GPUメモリ使用量
- コンテナ別GPU利用率
- 同時接続数
- 推論キュー長

これらをWeb UIおよびCLIから確認できることを必須とする。

## 13. セキュリティ設計

### 13.1 初期状態

- SSHは鍵認証を基本とする
- API外部公開は無効とする
- rootログインは無効とする
- 不要サービスは無効化する

### 13.2 実行分離

- AIアプリは原則コンテナ内で実行する
- デバイスアクセスは必要最小限に限定する
- モデル格納領域と設定領域を分離する
- 機密モデル利用時はオフライン運用を選択可能とする

### 13.3 更新

- OSイメージは署名検証可能であること
- 標準コンテナイメージは署名済み配布を推奨する
- ロールバック時に前版整合性を保てること

## 16. 技術的に見送る項目

以下は魅力的だが、v1必須要件には含めない。

- OS横断の汎用VRAMデデュープ
- 全ベンダーNPUの単一イメージ完全最適化
- Kubernetes前提のフル分散学習機能
- GUI前提の汎用デスクトップ機能
- ホストOS上への巨大MLフレームワーク全面常駐

見送り理由は、保守負荷、互換性、ハードウェア差分の大きさ、v1の価値との不一致にある。

## 17. 成功指標

本OSが成功していると判断するための主要指標を以下とする。

- 初期導入から推論開始までの時間が短いこと
- ドライバ起因のセットアップ失敗率が低いこと
- コンテナ更新時の再現性が高いこと
- 同一ハードウェアでの性能再現性が高いこと
- ローカルAI利用の立ち上がり工数を削減できること

## 18. 結論

AI特化Linux OSの本質は、機能を増やすことではなく、AI実行に必要な要素だけを正確に残し、不要な汎用性を削ることである。

したがって、v1で目指すべき姿は次の通りである。

- 壊れにくいOS
- GPU/NPUを活かせるOS
- AIコンテナを安全に回せるOS
- ローカル推論をすぐ始められるOS
- 運用者が状態を把握しやすいOS

この方針により、本OSは「LinuxベースのAIアプライアンス」として成立する。
