# AI特化Linux OS ローカル運用・内蔵オーケストレーション仕様

- 作成日: 2026-04-09

- 対象読者: プラットフォーム開発、SRE、ローカルAI利用者、プロダクト企画

## 1. 目的

本仕様は、AI特化Linux OSに「ローカルで簡単に使え、必要になったらそのまま複数ノードに広げられる」内蔵オーケストレーション機能を追加定義するものである。

目標は次の通りとする。

- 1台のPCやワークステーションで、数分以内にAIサービスを起動できること
- 同じ宣言ファイルを使って、家庭内ラボ、小規模オフィス、部門内サーバーへ拡張できること
- Kubernetesや外部コンテナ基盤を必須にしないこと
- モデル、GPU、文脈、キャッシュをAI向けに扱えること

## 2. 設計原則

- Single-node first
- Zero-conf by default
- Declarative but human-readable
- Local-first security
- Upgrade without losing context
- AI-specific resources as first-class citizens

## 3. 中核コンポーネント

本機能は以下の4コンポーネントで構成する。

### 3.1 `aiosd`

ホスト常駐のローカル制御デーモン。PID 1 配下の systemd サービスとして起動する。

責務:

- ノード初期化
- ハードウェア自動検出
- モデルキャッシュ管理
- ワークロード配置
- GPUスライス予約
- 文脈スナップショット管理
- ローカルAPI公開
- 追加ノード参加管理

### 3.2 `aictl`

人間向けの管理CLI。Kubernetesのような重い概念を避け、Docker Composeに近いわかりやすさを持つ。

主コマンド:

- `aictl init`
- `aictl doctor`
- `aictl apply -f stack.yaml`
- `aictl up`
- `aictl down`
- `aictl ps`
- `aictl logs`
- `aictl model pull`
- `aictl context ls`
- `aictl node pair`
- `aictl node join`
- `aictl upgrade plan`
- `aictl recipe run`

### 3.3 `Stack` マニフェスト

AI向けサービス群を記述する宣言的フォーマット。Kubernetesより少ない概念で、ComposeよりAI向けに拡張する。

扱う資源:

- 推論サービス
- 埋め込みサービス
- RAG補助サービス
- Web UI
- モデルバンドル
- モデルキャッシュ
- GPUスライス
- 文脈継続
- Fabric配置ポリシー

### 3.4 `AI Local Console`

localhost専用のWeb UI。CLIが使えない利用者でも起動、停止、監視、モデル選択、レシピ実行ができる。

## 4. 単一ノード運用モデル

単一ノードでは、オーケストレーション状態はローカルSQLiteに保持する。外部DB、etcd、Kubernetes API Server は不要とする。

### 4.1 起動フロー

1. OS初回起動時に `aiosd` がハードウェアを検出する
2. GPU/NPU、RAM、VRAM、NVMe、CXLの有無をプロファイル化する
3. ローカルAPIとWeb UIを `127.0.0.1` へバインドする
4. 初期モデルキャッシュ領域を作成する
5. 標準レシピを利用可能状態にする

### 4.2 利用者体験

利用者は次のいずれかで開始できる。

- `aictl recipe run local-chat`
- `aictl apply -f stack.yaml`
- Local Console で「チャット」「RAG」「画像生成」テンプレートを選択

### 4.3 ローカル向け自動化

単一ノードでは以下を自動で行う。

- 空きGPU/VRAMに応じた既定モデルサイズの提案
- モデルの段階的ウォームアップ
- 一定時間アイドルなサービスの自動停止
- よく使うモデルのプリロード
- ローカルディスク残量に応じたキャッシュ削除

## 5. 複数ノードへの拡張

本機能は「ローカル運用がそのまま小規模クラスターへ伸びる」ことを重視する。

### 5.1 拡張方針

- 1台目は自動的に `seed node` になる
- 2台目以降はペアリングコードまたは短期証明書で参加する
- 3台程度までの部門内クラスターを軽量構成で扱う
- 大規模クラスター化が必要なら外部基盤連携へ移行可能とする

### 5.2 内蔵クラスタ状態管理

- 単一ノード時はローカルSQLiteを使用する
- 複数ノード化時は軽量レプリケーションモードへ昇格する
- 少数ノード向けであるため、重い外部依存は避ける
- クラスタ状態は、ノード、スライス、モデル配置、文脈スナップショット、ジョブ意図を保持する

### 5.3 ノード参加

ノード参加は以下の流れとする。

1. 管理ノードで `aictl node pair` を実行する
2. 一時的な参加トークンと接続先情報を生成する
3. 参加ノードで `aictl node join` を実行する
4. プロファイル検出とセキュリティ検査を行う
5. 許可されたロールを付与して参加完了とする

### 5.4 ノードロール

- `control`
- `gpu-worker`
- `cpu-worker`
- `storage-cache`
- `edge-gateway`

## 6. `Stack` マニフェスト仕様

### 6.1 目的

`Stack` は、AIサービス群を1ファイルで起動、停止、更新、複製できるようにする。

### 6.2 主な特徴

- 10個前後の主要フィールドで扱える
- AI向けリソースを直接書ける
- 単一ノードと複数ノードで同じ形式を使える
- ワークロード、モデル、ネットワーク、永続データ、文脈をまとめて定義できる

### 6.3 主要オブジェクト

- `Stack`
- `InferenceService`
- `ModelBundle`
- `FabricPolicy`
- `TenantClass`
- `Recipe`

### 6.4 スケジューリング方針

- デフォルトは `local-prefer`
- 複数ノード時は `latency-first` `cost-first` `isolation-first` を選択可能
- GPUスライスが取れない場合は、待機、縮退、拒否を選べる

## 7. レシピ機能

レシピは、よくあるAI構成を1コマンドで作るテンプレートである。

### 7.1 例

- `local-chat`
- `team-rag`
- `vision-lab`
- `transcribe-and-summarize`
- `private-coding-assistant`

### 7.2 要件

- レシピは宣言的マニフェストへ展開できること
- 実行前に必要リソースを見積もること
- モデル未取得時は自動取得確認を行うこと
- 起動後にヘルスチェックとURL案内を出すこと

## 8. AI向けオーケストレーション機能

### 8.1 モデルキャッシュの共有

- 同一ノード上の複数サービスでモデルキャッシュを共有できること
- 差分アダプタだけを切り替えられること
- 署名付きモデルは検証済みであることをキャッシュメタデータに保持すること

### 8.2 文脈継続

- サービス再起動時にセッション文脈を残せること
- ノード更新時に重要セッションを先に保護すること
- 再開期限、暗号化、エクスポート可否を設定できること

### 8.3 ローカルSLO

- 単一ノードでも `低遅延優先` と `大吞吐優先` を選べること
- バックグラウンドタスクが対話APIを潰さないこと
- 指定SLOを満たせない場合は事前に警告すること

### 8.4 自動縮退

ローカル環境では、リソース不足時に停止ではなく縮退で逃がす。

例:

- 72Bモデルを停止し、8Bモデルへ自動切替
- 画像生成解像度を下げる
- バッチ処理を延期する
- 背景埋め込み処理を夜間へ回す

## 9. Local Console 要件

- `127.0.0.1` に標準バインドすること
- 現在のノード構成をカード形式で表示すること
- 利用可能GPU、VRAM、モデル、稼働サービスを表示すること
- レシピ一覧からワンクリック起動できること
- 起動済みサービスのURL、SLO、モデルを見られること
- ドレイン、再起動、アップグレードを対話的に実行できること

## 10. CLI 要件

### 10.1 方針

- 出力は人間可読を優先する
- JSON/YAML出力も選べる
- 危険な操作は確認を求める
- ローカル操作を最短経路にする

### 10.2 代表コマンド

```powershell
aictl init
aictl doctor
aictl recipe run local-chat
aictl apply -f .\stack.yaml
aictl ps
aictl logs legal-assistant-prod
aictl model pull registry.example.internal/models/qwen3-72b-regulated:2026.04.0
aictl node pair --role gpu-worker
aictl node join --token XXXX-YYYY --manager 192.168.10.12
aictl upgrade plan --target-version 1.2.0
```

## 11. アップグレード設計

- 単一ノードでは、文脈スナップショットを先に作成する
- 複数ノードでは、ワークロードを順次ドレインする
- モデルキャッシュは可能な限り維持する
- 更新後に失敗した場合は前版へ戻し、文脈再開を試みる

## 12. セキュリティ設計

- ローカルAPIは既定で loopback のみ
- リモート管理は明示的有効化時のみ
- ノード参加トークンは短寿命にする
- 参加時にOSバージョン、署名、プロファイル整合性を確認する
- モデルの外部持ち出し可否は `TenantClass` で制御する

## 14. この機能が買収価値につながる理由

- 個人利用と企業利用の間の溝を埋める
- 「1台で動く」ことが導入障壁を大きく下げる
- そのまま部門内標準へ育ちやすい
- 運用が簡単なほど、上位のセキュリティ機能やSLO機能が採用されやすい
- インストール後すぐに価値が見えるため、営業・PoC・社内展開に強い

## 15. 結論

本OSにとって重要なのは、高度な分散制御を最初から押し付けることではない。まずローカル1台で「簡単で、速くて、壊れにくい」体験を作り、その延長で小規模オーケストレーションへ自然に伸ばすことが重要である。

この内蔵オーケストレーションは、その橋渡しを担う。
